# Batch 8 人工验收问题记录

## Issue 1: 前后台文章阅读量仍为 0

### 现象

人工验收第 1 项时发现：多篇文章在前台打开或刷新后，前台文章页和后台文章列表的阅读量仍然是 0。

同时，“DDD 实战完全指南”这篇文章的阅读量显示为 3，说明后端接口或数据库并非完全失效，更像是我之前烟测命中的文章有效，而手动浏览路径没有完整触发。

### 期望行为

普通访客打开已发布文章后：

- `visit_events` 写入一条带 `article_id` 的 `pageview` 明细。
- 对应 `articles.view_count` 增加 1。
- 再次刷新或重新打开页面后，前台和后台能看到累计阅读量增长。
- 站点 PV 只增加 1，不因为普通路由 PV 和文章 PV 重复统计。

### 初步定位

后端直接调用 `/api/v1/collect` 携带 `article_id` 时可以正确递增文章阅读量。

真正缺口在前台文章页：文章页只在 `route.params.slug` 变化时调用带 `article_id` 的埋点，直接打开文章页或刷新文章页时，首屏加载不会触发该逻辑。因此多数人工浏览路径不会产生文章阅读量。

另一个相关风险是：通用路由埋点会发送不带 `article_id` 的普通 pageview。如果文章页自己也发送带 `article_id` 的 pageview，同一次文章访问可能造成站点 PV 双算。

### 处理状态

已修复，详见 `04-fix-report.md`。

## Issue 2: 前台阅读量比后台稳定少 1

### 现象

人工验收第二次发现：某篇文章前台显示阅读量为 7，后台显示为 8。

用户确认其中一次刷新后，后台阅读量增长了，但前台阅读量没有同步增长；之后再次刷新时两边一直差 1。

### 期望行为

普通访客打开文章页后：

- 后端 `articles.view_count` 增加 1。
- 当前前台文章页显示值与后端最新累计值保持一致。
- 后台文章列表读取数据库值时，不应与前台稳定差 1。

### 定位

这不是后端计数失败，而是前台显示链路使用了计数前的旧快照。

文章头部 `PostHeader.vue` 独立通过 SSR `useAsyncData` 拉取文章详情并显示 `view_count`。文章阅读量递增发生在客户端 mounted 后，因此页面初始显示的是“递增之前”的值；后台列表读取数据库时已经是“递增之后”的值，所以会出现前台比后台少 1。

### 处理状态

已修复，详见 `04-fix-report.md`。

## Issue 3: 后台访问日志默认进入时报“获取访问日志失败”

### 现象

人工验收第 3 项时，进入后台访问日志页面，页面提示“获取访问日志失败”。

### 期望行为

进入后台访问日志页面时，即使不填写任何筛选条件，也应正常加载第一页访问日志列表。

### 定位

后端 `/api/v1/admin/stats/visits?page=1&page_size=20` 默认列表请求触发 SQL 语法错误。

根因是 Batch 8 新增访问日志筛选时，SQL 拼接少了一个空格，生成了：

```sql
where 1 = 1order by created_at desc, id desc
```

PostgreSQL 报错：

```text
trailing junk after numeric literal at or near "1order"
```

之前自动测试覆盖了带筛选条件的访问日志查询，但漏掉了“无筛选条件默认进入页面”的路径。

### 处理状态

已修复，详见 `04-fix-report.md`。

## Issue 4: 文件列表使用状态全部显示未使用

### 现象

人工验收文件列表时发现：文件列表里的“使用状态”全部显示为“未使用”。

但部分图片已经能在前台正常显示，例如站长形象、站长头像、评论区图片等。

### 期望行为

文件如果已经被真实业务内容引用，应在后台文件列表显示为“使用中”。

至少应覆盖：

- 文章封面、文章正文中的 `/uploads/**` 图片。
- 站点设置中的站长头像、站长形象等 `/uploads/**` 图片。
- 评论内容/头像中的 `/uploads/**` 图片。
- 反馈内容、用户头像、友链头像/截图、动态内容中的 `/uploads/**` 图片。

### 定位

文件表 `files.status` 是上传时写入的静态字段，默认是 `0`。

前台图片能显示，只证明文件落盘和 `/uploads/**` 访问链路可用；后台文件列表原先直接读取 `files.status`，没有根据业务表中的真实引用关系计算“是否使用中”。

因此，已经被设置、文章、评论等内容引用的文件，仍会显示为“未使用”。

### 处理状态

已修复，详见 `04-fix-report.md`。
